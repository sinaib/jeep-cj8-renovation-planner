What Is This

AI-powered local restoration planner
for a 1989 Jeep CJ8 Scrambler

🔧 23 tools
🤖 2 AI agents
🧠 5 memory stores
💾 Local-first
Local-first: all data lives on disk in your home directory — no cloud sync, no accounts, no subscriptions. The AI agents talk to Anthropic's API, but your entire restoration plan (tasks, parts, costs, notes, decisions, research) is stored in plain JSON files you own forever.

Agentic: tell the agent what you want in plain language. It builds out phases, adds tasks, estimates Israeli market prices, saves technical procedures, flags risks, and searches the web — all automatically, without you filling in any forms.
How a Message Travels

Architecture Flow

Every message you type passes through a two-tier routing system before any AI responds. The router makes a fast decision about which agent can handle the request, minimizing cost while preserving full capability when needed.

↓

↓

↓

Routing Rules

Message contains image attachment FULL
Message longer than 120 characters FULL
"add", "search", "explain", system names, tool keywords FULL
"hi", "thanks", "what's next", "status", "how" + short FAST
Message shorter than 60 characters (no tool keywords) FAST
All other messages FULL

💬 Conversation Compression

After 12 messages, the Fast Agent compresses the oldest 8 into a summary. Nothing is lost — the full plan is always in the system prompt regardless. Saves tokens on long sessions without sacrificing memory of what's been discussed.

⚡ Prompt Caching

The static system prompt (~70% of all tokens) — CJ8 factory specs, Battlefield manual, JEEPOLOG data, supplier list, agent rules — is marked for Anthropic's cache. Cache hits cost ~10% of normal. Applies to every Full tier request after the first in a session.

What the Agent Remembers

The Intelligence Layer — 5 Memory Stores

The agent doesn't rely on conversation history alone. Five persistent stores accumulate knowledge about this specific Jeep over every session, shaping every future suggestion.

🚗
Car Facts

Specific truths about this Jeep. Grows as the agent learns. Shapes every future recommendation. Includes: engine_condition, mileage, diy_skills, goals, budget, known_issues, and more. Set via set_car_fact.

⚖️
Decisions

Permanent record of what was agreed. Organized by category: budget · safety · approach · scope · timeline · supplier. The agent refers back to these before making conflicting suggestions.

🔍
Research Notes

Findings from web searches and manual reading. Each note has a topic, finding, source, and optional task link. Prevents the agent from re-searching the same things across sessions.

⚠️
Gaps

Identified risks or missing work items. Three severity levels: critical · warning · suggestion. Dismissible. Can be converted to tasks. Surfaced by background analysis automatically.

🔗
Dependencies

Mechanical ordering constraints. Which tasks must happen before others and why. Enforced by the UI — blocked tasks are visually distinct. Set via set_task_dependency.

Task Anatomy

Task: name · system · phase · priority
  ├─ status:           flagged | todo | active | done | skipped
  ├─ parts[]:          name · estimatedCostILS · partNumber · purchased
  ├─ steps[]:          CJ8-specific how-to instructions (torque specs, gotchas)
  ├─ notes[]:          timestamped observations and measurements
  ├─ estimatedCostILS: Israeli market price estimate
  ├─ actualCostILS:    what you actually paid (set when marking done)
  └─ agentRationale:   why this task exists and why now
What the Agent Sees

Smart Context Selection

The Full Agent doesn't receive your entire restoration plan on every message — that would be slow and expensive. A context selector decides what's relevant and sends only that slice.

  taskId provided?  ────────────────────────→  Focused Task Detail
       │                                         (all fields, parts, steps, notes)
       ↓ no
  phaseId only?  ──────────────────────────→  Full Phase View
       │                                         (all tasks in that phase, detailed)
       ↓ no
  system keywords?  ───────────────────────→  System-Filtered Tasks
       │  "engine", "brakes", "fuel", etc.       (tasks for that system only)
       ↓ no
  cost keywords?  ─────────────────────────→  Cost Summary by Phase
       │  "cost", "budget", "₪", "spend"         (estimates + actuals, no steps)
       ↓ no
  Default  ────────────────────────────────→  Compressed Plan
                                               (phase names + task counts + status)

  ─────────────────────────────────────────────────────────────────────────
  Always appended regardless of mode:
    Car Profile · Recent Decisions · Active Gaps · All task/phase IDs
~50–80% fewer tokens per request compared to always sending the full plan. On a 50-task plan, default mode sends ~800 tokens of context vs. ~4,000 for the full plan. At Sonnet pricing, this is the single biggest cost saver in the stack.
What Runs Without You

Background Intelligence — 3 Silent Agents

Three analysis passes run automatically in the background — no prompting needed. Each is rate-limited and uses the cheapest effective approach. They write to the store but never produce visible chat messages.

🔄 On Plan Change SILENT
Trigger: any change · debounced 8s · max once per 10 min
Does: Scans for CJ8-specific gaps not yet in the plan — known failure points for this chassis, missing safety steps, mechanical dependencies that aren't captured
Tools: flag_gap only. No tasks added. No chat output.
Note: Browser-side only — runs while the app tab is open
✅ On Task Completion SILENT
Trigger: task marked 'done' · only if it unblocks dependents · 3s delay
Does: Populates parts + steps for newly-unblocked tasks. Adds watch-out notes from the completed task's context (e.g. "now that you've done X, watch for Y on Z")
Tools: set_task_steps · add_part_to_task · add_task_note
Note: Only fires if dependent tasks exist and are unblocked by this completion
📅 Weekly Health Check SILENT
Trigger: app load · max once per week · tracked in localStorage
Does: Flags "active" tasks with no recent notes (stale). Flags overloaded phases (>10 tasks). Flags missing restoration phases that a CJ8 should have.
Tools: flag_gap · add_task_note
Note: Runs on page load — fires even if you haven't used the app in a week
Where Everything Lives

Data & Storage Architecture

Disk Layout (via Vite plugin)

data/
  project.json         ← Full Zustand state
                         (written on every change)
  changelog.ndjson     ← Append-only event log
                         (every change, forever)
  snapshots/           ← Auto-saved before bulk
    2026-03-14T15-52-26.json    agent changes
    2026-03-13T09-11-04.json
    ...
💾
Disk JSON
The real persistent record. POST /api/project on every change. Survives browser wipes, cache clears, reinstalls. The canonical source of truth.
🔵
localStorage
Instant sync backup. Written synchronously. Read on load as fallback if disk is unavailable. Also stores: conversation history, weekly health check timestamp, UI preferences.
🗄️
IndexedDB
Binary file data — photos, PDFs. Never stored in JSON. Metadata (filename, taskId, annotation) lives in Zustand state; the raw bytes live here.
⚡
Zustand
In-memory reactive state. Source of truth at runtime. All UI reads from here. Mutations write to both localStorage and disk asynchronously.
